HiveMatrix — Capabilities & Setup

What it can do, how the pieces fit, and how to set it up — including a full tour of the Settings screen.
Updated 2026-06-25 · reflects the Bee→Lane naming (agents call lane-native tools like desktop_action) and the live configuration on this Mac.
HiveMatrix is a personal AI operations system. A headless daemon on your Mac runs tasks and long-running directives across a fleet of AI models — frontier (Claude, Codex/ChatGPT), a local model (Qwen), or a router that mixes them — and gives those agents real capabilities through capability lanes: web search, browser automation, and desktop control. You drive it from the desktop console or the iPhone app, locally or over a Cloudflare tunnel.

1The big picture

Daemon

A headless background service (launchd, auto-restarts) bound to 127.0.0.1:3747. It owns tasks, directives, the model router, connectivity policy, approvals, memory, and the capability lanes. Everything else is a client of it.

Console & iPhone app

The desktop console (served by the daemon) and the HiveMatrix-iOS app are both thin clients. The iPhone app reaches the daemon over a Cloudflare tunnel; both show the same board, tasks, directives, and settings.

Model plane

Three harnesses behind one contract: Claude Code, Codex, and Qwen (local). A role-based router picks which one runs a given piece of work.

Capability lanes

Embedded capability lanes the agents can call: Browser Lane (web fetch/search + authenticated browser), Desktop Lane (desktop), plus internal Memory Lane (memory) and Manager Lane (coordination).

Each task picks a model. Beyond the individual models, three macro run modes decide the local-vs-cloud split:

Set any of these as the default in Settings, or pick per-task in the New Task form.

3Capability lanes

Capability lanes are how an agent does more than think. As of now they're wired into the local (Qwen) agent's tool loop too — so the local model can search the web, drive a browser, or control the desktop, each gated by the current connectivity mode.

LaneDoesAgent toolStatus here
Browser Lane One browser/web lane: read-only fresh web retrieval with citations (current facts, news, prices, docs) and authenticated / stateful browser workflows (login, multi-step, forms, screenshots) — the authenticated path runs as a Codex Computer Use task. With no Codex auth, an opt-in Desktop Lane fallback (browserLane.desktopFallback) lets the local model drive a desktop browser instead. The COO routes to Browser Lane; the lane picks the backend. hivematrix_browser embedded · running
Desktop Lane Native macOS automation via a signed Swift helper: AppleScript → Accessibility actions → click/type as last resort desktop_action helper on :3748
Memory Lane Memory compilation & retrieval (internal; launch-agent lane, toggle in Settings → Capability Lanes) — stopped
Manager Lane Planning / routing / review coordination (internal) — planned
Availability follows connectivity. The Browser Lane web/authenticated modes need the internet, so they're disabled in local-only and offline modes; Desktop Lane works in every mode. An agent is only shown the lanes it can actually use right now. Authenticated browser work without a Codex subscription: by default the lane refuses (it needs the Codex Computer Use backing). Set browserLane.desktopFallback: true in ~/.hivematrix/config.json to let it degrade to a local Desktop Lane-driven browser instead — lower reliability, but it keeps authenticated browser work running with no subscription.
Message Lane (SMS/iMessage). Two-way. Turn on the iMessage channel (Settings → Message Lane) and grant Full Disk Access so the in-daemon poller can read Messages chat.db: inbound texts from allowlisted senders route to a reply or a new task, and the agent sends back out via message_send (allowlisted recipients only). Non-allowlisted handles are ignored inbound and refused outbound.

4Connectivity modes

The daemon always operates in one of three modes (auto-detected, or set manually in Settings → Connectivity):

cloud-ok

Default. Frontier APIs reachable; every capability available.

local-only

LAN/loopback only. Frontier + Browser Lane off; Qwen + Desktop Lane on.

offline

No network. Only the local model + Desktop Lane on local apps.

Auto-degrades when frontier usage windows are exhausted (→ local-only) or the network is unreachable (→ offline), and restores automatically. A manual override always wins.

5Tasks

A task is one unit of work for an agent. Create one from the board (＋ New task):

  • Instructions — what to do (title is optional; it's derived from the instructions if blank).
  • Project — pick a discovered project and the working directory is filled in automatically; the agent then knows where to operate. ★ marks your active project.
  • Model — any option from §2.
  • Attachments — click ⊕ Browse files to attach local files; they're passed to the agent.

Each task card supports quick archive (the ⌫ on hover) without opening it. Open a task to see the live transcript and actions:

  • Retry / Archive / Cancel / Delete depending on state.
  • Reply — when an agent needs input it enters needs_input and shows its question with a reply box; type an answer and the agent resumes (works in the console and on iPhone).

6Directives

A directive is a standing objective that runs over time — a goal, optional success criteria, and a trigger policy (manual, or a repeat interval like PT4H / P1D). The daemon plans tasks toward the goal, executes them (honoring your default run mode, including Cloud-only), and tracks proven criteria in a run journal. Create one from the right-hand ＋ New directive panel.

7Skills & commands

Named, reusable recipes you can launch from the Run command / Run skill panel (or later, by typing /name). They come in four flavors, sorted by where they live and whether an LLM is in the loop:

FlavorWhere it livesShapeRun result
cmd
flat slash command		 ~/.claude/commands/<name>.md
(subdirs namespace to ns:name)		 Single markdown file: optional frontmatter (description, argument-hint, allowed-tools, model) + a prompt body. Nothing else on disk. Creates a task on the board; the model follows the prompt.
folder
folder skill (bundle)		 ~/.claude/skills/<name>/SKILL.md
+ sibling files (scripts, refs)		 A SKILL.md plus any files it needs at runtime. The bundle travels together, so the recipe can call ./pull.sh, read a references/*.md, or ship a template. Creates a task on the board; the agent can execute the bundled files by relative path.
skill
brain instruction skill		 ~/_GD/brain/skills/<name>.md
(kind: instruction)		 A distilled recipe with frontmatter (name, description, tags, compat, trusted) + a body that says when and how to apply. Discoverable by every model/harness; trusted skills are auto-shown to agents at run time. Creates a task on the board; an agent applies the recipe (and calls skill_used so it earns its keep).
ops
brain script skill		 ~/_GD/brain/skills/<name>.md
(kind: script, interpreter: bash | node | python3)		 Same on-disk shape as a skill, but the body is a deterministic script. No model in the loop. Only runs when trusted (imported ops stay untrusted until you approve). Runs in-panel with a live log — no board task, no LLM call. Same input → same output, every time.
Two panels, two homes. The console has a Run command picker for your local Claude profile (cmd, folder) and a Run skill picker for the HiveMatrix brain library (skill, ops — grouped as "Skills" and "⚙ Ops / scripts"). Import a folder skill into the brain library from the command panel if you want every model/harness to see it, or want to distill it into a shared recipe.

When to use which — samples

cmd — a prompt you already trust

Best when the recipe is just words and you already invoke it as a slash command in Claude Code. No bundled assets, no determinism guarantee — the model reads the prompt body and does the work. Frontmatter is optional; a bare markdown file with the prompt is a perfectly valid cmd.

---
name: import-all
description: Run all daily data imports in sequence
argument-hint: "(no args)"
---
Run each step below sequentially: StoreMaster SharePoint,
Vodafone, Store GPS, Meta Raw, then Meta Raw 7-Day.
<body: the prompt the agent follows>

Samples: /import-all (daily data pull), /pushall (flush every repo), /topemail (recent mail across accounts) — each is a single .md under ~/.claude/commands/. Subdirs namespace to ns:name (e.g. ~/.claude/commands/hive/deploy.md → /hive:deploy).

Reach for it when: the recipe is short, model-driven, and doesn't ship helper files. If the prompt already exists in your Claude Code profile, that's your cmd — no port needed.

folder — a recipe that ships with files

Best when the recipe needs a helper script, a long reference, or a template at runtime. The folder bundles them so the agent finds them at a known relative path.

Sample: /knox-cli — a SKILL.md that instructs the agent when to use knox tools, alongside supporting docs (~/.claude/skills/knox-cli/). Bundled files travel with the recipe.

Reach for it when: the recipe references files by path, or a helper script would be duplicated if you inlined it.

skill — a durable playbook every agent should know

Best when the recipe is a reusable pattern HiveMatrix should apply across projects — usually distilled from a directive retrospective. Trusted skills are injected into agent context automatically so agents follow them without being asked; imported (untrusted) ones stay dormant until you approve them in the Run skill panel.

---
name: bible-study
description: Use when studying a Bible passage…
kind: instruction
compat: all
tags: [study, reference]
trusted: true
---
<body: when to apply, and how>

Samples: bible-study, calendar, weekly-ai-roundup — each is a body of instructions the agent applies when the description matches the task. Harness-agnostic (compat: all) means Claude, Codex, and local Qwen can all use them; useCount ticks up when the agent calls skill_used, so unused skills surface for pruning.

Reach for it when: multiple models/harnesses should apply the same judgment, or the recipe should surface itself when relevant (not just when invoked). Fastest path to author one: run a directive, then ask HiveMatrix to distill the retrospective into a skill.

ops — a deterministic operation, no model needed

Best when the "how" is fixed and you want the same output every time: releases, backups, health checks, integration proofs. No LLM in the loop — just an interpreter running a script the operator can read. The body is the program; HiveMatrix pipes it to bash, node, or python3 and streams the log into the panel.

---
name: hive-run-all-proofs
description: Run every HiveMatrix proof/health check…
kind: script
interpreter: bash
compat: all
trusted: true
---
set -uo pipefail
cd "${HIVEMATRIX_REPO:-$HOME/hivematrix}"
<body: the exact commands, in the exact order>

Samples: hive-run-all-proofs, hive-qwen-readiness, hive-scope-wall, hive-verify-release, release-hivematrix — every one is a bash script you can read line-for-line before running. Nothing gets improvised at run time.

Trust matters more here. An instruction skill just tells an agent what to consider; an ops script runs on your machine. Imported ops land trusted: false and are refused by the runner until you approve them in the Run skill panel — read the body first, then flip the switch.

Reach for it when: you'd be annoyed if the model reinterpreted the steps — you want the exact commands, in the exact order, with a diffable log and a clean exit code you can gate a release on.

Picking one over another — head-to-head

If you want…Reach forNotWhy
A prompt you already invoke as /name in Claude Code cmd skill A skill wouldn't add anything the flat prompt file doesn't — and would need a second copy in the brain library.
The recipe to ship a helper script, template, or long reference folder cmd A single .md can't bundle files; the folder lets the agent call ./pull.sh or read references/api.md by relative path.
Every model/harness (Claude, Codex, Qwen) to apply the same judgment skill cmd / folder cmd/folder live in ~/.claude/ and are Claude-profile only; a brain skill is harness-agnostic (compat: all).
An agent to self-select the recipe when the task matches — without being told skill (trusted) cmd / folder Trusted skills are surfaced to the agent by description; cmd/folder only fire when the operator picks them.
Deterministic, diffable execution — same output every time, gating a release ops skill An LLM might reinterpret the steps. An ops script is the exact commands, in the exact order, with a clean exit code you can gate CI on.
A quick throwaway "run this prompt once" — no need to save it a plain task cmd Type the prompt into the task composer. A cmd is worth it only when you'll run the same recipe again.
Rule of thumb. Needs judgment → skill (or cmd / folder if it's already a Claude Code slash command). Needs determinism → ops. Ships with files → folder. None of the above → the flat cmd.
Where do launched tasks show up? cmd, folder, and instruction skill launches create tasks on the board. If you haven't picked a project, they land under the synthetic ops project bucket, so a filtered board can look empty — the launch toast will call this out (e.g. "Launched /import-all in ops — current board filter is digibot"). Script ops runs don't create a board task; watch the in-panel log instead.

8Remote access (iPhone)

The daemon is loopback-only; cloudflared bridges it to the internet when you choose.

  1. Settings → Remote Access → Start tunnel for a quick test URL, or use a named tunnel (Advanced) for durable, multi-user access.
  2. A pairing QR appears — open HiveMatrix on iPhone and scan it. It encodes the URL + access token, generated locally; the token never travels through Cloudflare.
  3. Or paste the access token manually (remote visitors never get it auto-injected).
For anything lasting, use a named tunnel. Quick trycloudflare.com URLs are random and change on restart. See the companion brain doc "HiveMatrix — Cloudflare Tunnels & Remote Access" for the full quick-vs-named breakdown and the multi-user setup.

9First-time setup

Onboarding (right-hand Setup panel) tracks these. In order:

1 · Configuration file ~/.hivematrix/config.json holds model + provider settings. Created on first run.
2 · Local model (Qwen) Install LM Studio, load qwen/qwen3.6-27b (MLX 8-bit) and start its server on :1234. HiveMatrix probes readiness (tool-calling, decode rate) before using it. Pin the model with no TTL and set LM Studio to launch at login for always-on.
3 · Background daemon (launchd) Installs ~/Library/LaunchAgents/com.hivematrix.daemon.plist so the daemon runs 24×7 and restarts on exit.
4 · Brain memory plane Canonical memory lives at ~/_GD/brain. Agents read/curate it for durable context.
5 · Frontier model access Sign in to the claude and/or codex CLIs (subscriptions). Detected automatically and shown under Settings → Backends.
6 · Desktop Lane (desktop control) Grant the Swift helper Accessibility and Screen Recording in System Settings → Privacy & Security. (Rebuilding the helper can invalidate a prior grant — re-toggle it if the Desktop Lane shows no permission.)

10The Settings screen

Open with the ⚙ gear (top of the console). Four tabs:

Models tab

  • Default model — the model new tasks use unless overridden (set it to Cloud-only for a no-local posture, or Mixed for the blend). Save default.
  • Backends — live detection of Local (LM Studio/Qwen), Claude CLI, and Codex CLI, with a ✓ when configured.
  • Local server endpoint — the OpenAI-compatible URL for the local model (default http://localhost:1234/v1).
  • Appearance — theme (System / Light / Dark) and an optional wallpaper (path or upload).
  • Remote Access — tunnel status, Start/Stop, the public URL, pairing QR, the access token, and the Advanced named-tunnel form. (See §8.)

Projects tab

Projects auto-discovered from git repos, Claude Code history, and VS Code recents. Click one to make it the active board filter; ↻ Re-scan refreshes. ★ = the pre-selected active project whose path pre-fills New Task / New Directive.

Capability Lanes tab new

A live view of every capability lane: a status dot (green = running & healthy, amber = running but unhealthy, grey = stopped), the runtime mode (embedded = runs in the daemon and follows connectivity; launchagent = separate service; planned = not yet wired), and health badges. Launch-agent lanes (e.g. Memory Lane) have a Turn on / Turn off toggle that installs or removes their macOS LaunchAgent. Embedded lanes can't be toggled — they follow the connectivity mode. The same view exists in the iPhone app under Settings → Capability Lanes.

Connectivity (right panel)

Shows the current mode and lets you force cloud-ok / local-only / offline (or return to auto). Useful to go private (local-only) or to test offline behavior.
