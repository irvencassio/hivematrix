Cloudflare Tunnels & Remote Access

How HiveMatrix gets a public URL · quick vs. named tunnels · whether it holds up for multiple users
Written 2026-06-12 · Reflects the live state of this Mac at time of writing.
TL;DR. Yes — prot-crafts-cameron-tomato.trycloudflare.com is a randomly generated, account-free “quick tunnel.” Cloudflare hands these out with no signup, which is why it “just worked.” But a quick tunnel is a throwaway dev URL: it changes on every restart, has no uptime guarantee, is rate-limited, and is not meant for real users. For long-term or multi-user use you want a named tunnel tied to a Cloudflare account + your own domain — and you already have one of those running on this machine. The fix is to stop relying on the quick tunnel and standardize on the named tunnel (plus Cloudflare Access for auth).

1 · The two kinds of Cloudflare tunnel

Both are run by the same cloudflared binary, but they are completely different products. HiveMatrix can drive either one (Settings → Remote Access).

Why no signup felt suspicious — and is fine for testing. Cloudflare runs the *.trycloudflare.com service as a free, anonymous on-ramp to cloudflared. It is genuinely zero-config. It is also genuinely disposable: treat any trycloudflare URL as valid only until the next restart.

3 · Will this hold up long-term with multiple users?

Quick tunnel — No. Don’t build on it. The URL rotates on restart, it’s rate-limited, has no uptime guarantee, and puts zero auth in front of the daemon (only HiveMatrix’s bearer token stands between the internet and your machine). Fine for a demo; wrong for users you can’t personally re-send a link to.
Named tunnel — Yes, with the right shape. A named tunnel on a free Cloudflare account gives you a stable hostname, real uptime, and the ability to put Cloudflare Access in front (email OTP / SSO / allow-listed identities) so each user authenticates at the edge before a request ever reaches the daemon. That is the correct multi-user posture.

The honest caveat: one Mac is still one origin

Even with a perfect named tunnel, every request ultimately lands on this one laptop’s daemon (127.0.0.1:3747, single SQLite database, single process). Cloudflare makes it reachable and authenticated; it does not make it highly available or horizontally scalable. For a handful of trusted users that’s fine. Before it’s “real product for many users,” the origin — not the tunnel — becomes the limiting factor (always-on host, backups, per-user identity inside HiveMatrix, and eventually a server rather than a laptop).

4 · Recommended setup for durable / multi-user access

  1. Create a named tunnel (you effectively already have one). In the Cloudflare Zero Trust dashboard: Networks → Tunnels → Create, choose “Cloudflared,” copy the connector token.
  2. Route a hostname to the daemon: add a public hostname hive.yourdomain.com → http://localhost:3747 on that tunnel.
  3. Run it from HiveMatrix: Settings → Remote Access → Advanced: named tunnel → paste the connector token + hostname (this calls POST /tunnel/start-named). Or let the system cloudflared service run it (what’s happening today).
  4. Put Cloudflare Access in front: add an Access application for that hostname with an email-OTP or SSO policy. Now strangers can’t even reach the login.
  5. Keep the bearer token as defense-in-depth: HiveMatrix still requires its access token on every route except /health. The console never hands the token to tunneled visitors — they paste it manually (from local Settings) or scan the pairing QR.
  6. Stop using quick tunnels for anything you expect to revisit.

5 · Security model (already enforced)

  • The daemon binds loopback only (127.0.0.1:3747); cloudflared is the only thing bridging it to the internet.
  • Bearer-token auth on every route except /health, /, /console.
  • Remote (tunneled) visitors — detected via the CF-Connecting-IP header — are served the console without the token baked in; they must paste it or pair by QR. Loopback users get it injected automatically.
  • Pairing QR (GET /tunnel/qr) encodes {url, token} and is rendered locally with qrencode — the token never traverses Cloudflare.
