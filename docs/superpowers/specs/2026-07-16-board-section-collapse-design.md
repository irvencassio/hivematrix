# Board Section Collapse/Expand — Design

## Problem (operator ask)

The left sidebar (`section.col.board`) stacks the nav buttons, the New Task form, and
the full Board (QUEUED/IN PROGRESS/REVIEW/etc. lane cards) in one fixed-width column.
The Board lane container has no way to get out of the way — an operator who wants to
glance at Chat/Memory/Roles/Tools/Goals or just use the New Task form without the
lane cards eating vertical space has no option today. Ask: a collapse/expand toggle
next to the "Board" heading that hides/shows the lane container, persisted across
reloads.

## Current state (verified against HEAD `dbb7dd71`)

Single source of truth: `src/daemon/console.ts`, a server-rendered `CONSOLE_HTML`
string (no build step). Relevant markup, `console.ts:1882-1885`:

```html
<div id="boardSec" class="board-sec">
  <div class="board-sec-header">Board <span id="archiveBtn" class="archive-link" onclick="archiveCompleted()" title="Archive review/done/failed tasks"></span></div>
  <div id="board"></div>
</div>
```

- `.board-sec` / `.board-sec-header` CSS: `console.ts:691-692` — header is
  `display:flex; align-items:center; gap:8px`, no existing collapse hook.
- `#board`'s content is generated by `renderBoard()` (`console.ts:2436-2467`), which
  runs on a 5s poll (`setInterval(refresh, 5000)`, `console.ts:10029`) and after most
  task-mutating actions. Critically, `renderBoard()` only ever touches
  `document.getElementById("board")` (sets `.innerHTML`) and
  `document.getElementById("archiveBtn")` (sets `.textContent`) — it never reads or
  writes anything on `#boardSec` itself. A collapse mechanism keyed off a class on
  `#boardSec` therefore survives every periodic re-render undisturbed; one keyed off
  `#board`'s own DOM (e.g. a data attribute) would not, since `innerHTML` assignment
  destroys it.
- **Existing analogous pattern 1** — the right context panel, `toggleContext()` /
  `wireCtxSections()` / the `applyCtxState()` IIFE (`console.ts:4662-4697`): toggles a
  class (`ctx-collapsed`) on `<main>`, flips a button's `.on` class, persists to
  `localStorage['hm_ctx_collapsed']`, and an unconditional (not gated behind
  `requireToken()`) top-level IIFE restores it on load. CSS: `main.ctx-collapsed
  .col.context { display: none; }` (`console.ts:180-182`) — this also rewrites
  `grid-template-columns`, because collapsing the *right* rail reclaims width for the
  center column. **Not reusable as-is**: that grid-column rewrite is specific to the
  3-column resizable layout; the Board sits in the fixed-width *left* column
  alongside other nav elements that must keep their width. Confirmed out of scope by
  the dispatch itself — do not touch `ctx-collapsed`/`<main>`'s grid logic.
- **Existing analogous pattern 2** — per-lane collapse, `getCollapsedLanes()` /
  `toggleBoardLane()` (`console.ts:2352-2361`): a `Set` of collapsed lane keys in
  `localStorage['hm_lanes_collapsed']`, read fresh by `renderBoard()` on every
  render (necessary because collapsing *inside* `#board` can't survive the
  `innerHTML` rebuild any other way). Good precedent for "collapse state as a
  localStorage-backed toggle next to a `▾`/`▸` caret" but scoped to individual lanes,
  not the whole section — collapsing every lane individually still leaves each lane's
  header row (name + count) visible per lane, which does not reclaim as much space as
  hiding `#board` outright and does not match the ask (a single control that hides
  the *entire* lane container).
- **Glyph convention**: `▾` (expanded, click to collapse) / `▸` (collapsed, click to
  expand) is used twice already — `.tools-caret` (`console.ts:7786`,
  `(open ? '▾' : '▸')`) and the per-lane `status-card-caret` inside `renderBoard()`
  itself (`console.ts:2454`, `(isCollapsed?'▸':'▾')`). Reused verbatim here for a
  third, consistent instance of the same affordance.
- Test-file boundary check: `taskFormSlice()` in `console.test.ts:2293-2298` slices
  `CONSOLE_HTML` from `id="taskForm"` to `id="board"` — it already includes today's
  `board-sec-header`/`archiveBtn` markup in that slice. Adding a new span before
  `archiveBtn` only adds content already inside that window; no existing
  `taskFormSlice`-based assertion does a whole-slice equality check, so nothing
  breaks.

## Approaches considered

**A. Toggle a class on `<main>` (reuse `ctx-collapsed`'s mechanism/key).** Rejected —
conflates two unrelated concerns (resizable right-panel width vs. a fixed-width left
column's internal section) and would require also touching the grid-column CSS the
dispatch explicitly said to leave alone.

**B. Extend `toggleBoardLane`'s per-lane Set to a "collapse all lanes" convenience
action** (no new key, no new markup — just call `toggleBoardLane` for every
`LANE_DEFS` key). Rejected: still renders each lane's header row (icon + name +
count) since `renderBoard()` always prints the lane head even when
`isCollapsed`(`console.ts:2452-2453`) — reclaims less space than hiding `#board`
outright, needs one Set entry per lane (N writes instead of 1), and conceptually
answers a different question ("which lanes are open") than what's asked ("is the
Board section open").

**C. (Recommended) New scoped toggle on `#boardSec` + a new `hm_board_collapsed`
localStorage key**, structurally mirroring `toggleContext`/`applyCtxState` (approach
A's pattern) but scoped to `#boardSec` instead of `<main>`, with no grid/width side
effects. One class, one key, reuses the already-established `▾`/`▸` caret glyph
convention. Smallest diff that satisfies the ask exactly: hides the whole lane
container, keeps the heading row (title + toggle + archive link) visible, persists,
restores on load.

## Recommended design

CSS — add immediately after the existing `.board-sec-header` rule
(`console.ts:692`):

```css
.board-toggle { cursor: pointer; color: var(--muted); font-size: 11px; user-select: none; }
.board-sec.collapsed #board { display: none; }
```

No transition — the dispatch marks animation optional and explicitly says an instant
`display:none` is fine if it keeps things simpler; adding a transition here would
mean animating a flex/block layout child's height, which is non-trivial with plain
`display:none` (needs a max-height/opacity dance) for a purely cosmetic gain. Skipped.

HTML — `console.ts:1882-1885`, insert one new `<span>` between "Board" and the
existing archive link:

```html
<div id="boardSec" class="board-sec">
  <div class="board-sec-header">Board <span id="boardToggle" class="board-toggle" onclick="toggleBoardSection()" title="Collapse Board">▾</span> <span id="archiveBtn" class="archive-link" onclick="archiveCompleted()" title="Archive review/done/failed tasks"></span></div>
  <div id="board"></div>
</div>
```

Default glyph is `▾` (expanded) so a first-ever page load with no stored preference
renders exactly like today (no regression).

JS — placed next to `getCollapsedLanes`/`toggleBoardLane`
(`console.ts:2352-2361`), i.e. grouped by *subject* (Board-area logic lives
together) rather than by *pattern* (next to the unrelated `ctx-collapsed` code) —
matches AGENTS.md's one-maintainer test: a future reader scanning "how does the
Board section work" finds this beside the other Board-collapse helpers it sits next
to in the UI, not four thousand lines away next to an unrelated right-panel concern:

```js
function toggleBoardSection() {
  const sec = document.getElementById('boardSec');
  if (!sec) return;
  const collapsed = sec.classList.toggle('collapsed');
  const btn = document.getElementById('boardToggle');
  if (btn) { btn.textContent = collapsed ? '▸' : '▾'; btn.title = collapsed ? 'Expand Board' : 'Collapse Board'; }
  try { localStorage.setItem('hm_board_collapsed', collapsed ? '1' : '0'); } catch (e) { /* ignore */ }
}

// Restore the persisted collapse state on load. #boardSec/#boardToggle are static
// markup (unlike #board's renderBoard()-generated innerHTML), so this only needs to
// run once. Deliberately a named function + explicit call (not a bare IIFE like
// applyCtxState) so it's unit-testable the same way toggleBoardSection is — a
// one-line style deviation, not a behavior difference.
function applyBoardSectionState() {
  try {
    if (localStorage.getItem('hm_board_collapsed') === '1') {
      const sec = document.getElementById('boardSec');
      if (sec) sec.classList.add('collapsed');
      const btn = document.getElementById('boardToggle');
      if (btn) { btn.textContent = '▸'; btn.title = 'Expand Board'; }
    }
  } catch (e) { /* ignore */ }
}
applyBoardSectionState();
```

Runs unconditionally at script-parse time (like `applyCtxState`), not gated behind
`requireToken()` — `#boardSec`/`#boardToggle` are static markup already in the DOM by
the time the inlined `<script>` runs, so no ordering hazard, and this avoids a
flash-of-uncollapsed-Board before any async auth check resolves.

## Non-goals

- Not touching `ctx-collapsed`/`<main>`'s grid-column logic — different concern
  (resizable right panel width), confirmed out of scope by the dispatch.
- No animation/transition — explicitly optional in the dispatch; a plain
  `display:none` keeps the diff smaller with no product cost.
- No DECISIONS.md entry — introduces no new persistent store, orchestration
  primitive, or product concept (kernel: Event/Task/Directive/Policy/Persona/Memory
  untouched). It's one more `localStorage` UI-preference key following the exact
  existing convention (`hm_ctx_collapsed`, `hm_lanes_collapsed`, `hm_sec_<id>`).
- Not changing `toggleBoardLane`/per-lane collapse — orthogonal, unaffected.

## Testing approach

Two tests in `console.test.ts`, following the two precedents named in the dispatch:

1. Execution-based round trip (mirrors `"getStoredView / setStoredView round-trip
   through localStorage..."`, ~line 2986): extract `toggleBoardSection` and
   `applyBoardSectionState`, run them against a hand-built fake `#boardSec`/
   `#boardToggle` element (a minimal `classList` + settable `textContent`/`title`)
   and a fake `localStorage`, and assert the full cycle — toggle collapses + flips
   the glyph + persists `"1"`, toggle again expands + flips back + persists `"0"`,
   and a fresh restore call with `"1"` already stored collapses a freshly-initialized
   (default, expanded) element without needing a click.
2. Structural regex assertions (mirrors the Snippets `loadSnippets`/`saveSnippets`
   test, ~line 3375, and the plan-header-template convention of asserting markup
   directly): the `<span id="boardToggle">` markup exists inside
   `board-sec-header` with the default `▾` glyph and wired to
   `onclick="toggleBoardSection()"`, and the CSS rule
   `.board-sec.collapsed #board { display: none; }` is present — the execution-based
   test above uses a fake `classList`/`localStorage` and so never actually proves the
   real CSS selector exists; this closes that gap.
